## Q1 · 退款的过账模型

- **背景**：平台订单可携带 `refunds[]`（含退款金额）。当前含任何退款的订单在提交阶段被整单拒。
- **待决问题**：退款一旦决定入账，应以何种记账模型表达？
- **可能的处理方向（不预设）**：负向销售记录 / 原单红冲 / 独立退款单据 / 其它——**由会计师确认**。
- **当前保守行为**：含退款订单整单拒（拒因码 `has_refunds`），退款不进账本。
- **影响面**：`has_refunds` 拒因；是否需要新的记录类型 / 列；与 `sales.payment_status` 的关系。

## Q2 · 部分退款的分摊

- **背景**：部分退款可能只退某些行或某个金额。
- **待决问题**：部分退款应按行分摊、按金额整单冲减，还是其它口径？
- **可能的处理方向（不预设）**：按明细行分摊 / 按订单金额整单处理 / 由平台提供的退款行明细直接映射——**由会计师确认**。
- **当前保守行为**：同 Q1，含退款即整单拒，不做任何分摊。
- **影响面**：明细行金额、表头 = Σ 明细 不变量、库存是否回补。

## Q3 · 退款的期间归属

- **背景**：退款发生日可能晚于原订单日期，跨会计期间。
- **待决问题**：退款记在退款发生日，还是冲减原单所在期间？
- **可能的处理方向（不预设）**：退款发生日 / 原单日期 / 结算日——**由会计师确认**（影响期间损益）。
- **当前保守行为**：不过账退款，无期间归属问题。
- **影响面**：期间损益、报表口径。

## Q4 · 平台费用的科目归属

- **背景**：平台可能收取手续费 / 佣金 / 广告费等（当前在预览中仅信息展示，`fees[]`）。
- **待决问题**：各类平台费用应归入哪个费用科目？是否按平台区分？是否可能被判为 COGS？
- **可能的处理方向（不预设）**：经营费用（细分类别）/ 与 COGS 的关系 / 按平台分设科目——**由会计师确认**（`is_cogs` 判定属会计红线，见 `CLAUDE.md`）。
- **当前保守行为**：平台费用不入账（预览仅展示 `fees` 合计）。
- **影响面**：`categories` 科目映射（红线）、费用报表。

## Q5 · 平台费用的确认时点

- **背景**：费用可能在订单级出现，也可能在结算单（payout / settlement report）级汇总。
- **待决问题**：费用按订单级确认，还是以平台结算单为准？是否需要引入结算单数据源？
- **可能的处理方向（不预设）**：订单级 / 结算单级 / 两者并存——**由会计师确认**。
- **当前保守行为**：不确认平台费用；无结算单数据源。
- **影响面**：是否需要新的结算单拉取 / 存储；与订单级数据的对账。

## Q6 · 运费的处理升级

- **背景**：当前买家付运费与卖家实际运费成本**双双不入账**（`shippingCost` 恒 0）。
- **待决问题**：买家付运费是否单列为收入？卖家实际运费成本如何记？
- **可能的处理方向（不预设）**：买家运费计收入 / 保持不入账 / 卖家运费计费用或 COGS——**由会计师确认**。
- **当前保守行为**：`shippingCost=0`，运费不入账；整单对账守卫把运费 + 运费税计入「刻意不过账」项。
- **影响面**：收入口径、`total_mismatch` 守卫公式的假设。

## Q7 · 退款税额与税含价店铺

- **背景**：退款可能含税额；税含价店铺（Shopify `taxesIncluded` / Woo `prices_include_tax`）当前被 `total_mismatch` 拒绝。
- **待决问题**：退款是否冲减销项税？是否随 EC6 一并支持税含价店铺（需拉取含税标志）？
- **可能的处理方向（不预设）**：冲减销项税 / 不冲减 / 税含价店铺单独立项——**由会计师确认**（VAT/GST/销项税口径属会计红线）。
- **当前保守行为**：不处理退款税额；税含价店铺整单拒（`total_mismatch`），不错误入账。
- **影响面**：税额列、`total_mismatch` 守卫、provider 是否需拉取 `taxesIncluded`。

## Q8 · 部分付款与结算币种

- **背景**：`PARTIALLY_PAID` 等状态当前被拒（需平台 transactions API 才知实付额）；`sales` 无 currency 列，MVP 单店铺币种。
- **待决问题**：是否引入部分付款映射（`payment_status='partial'` 已被 schema 支持）？payout 币种 ≠ 订单币种时如何折算？
- **可能的处理方向（不预设）**：引入 transactions API 取实付额 / 维持整单拒 / 结算折算口径——**由会计师确认**。
- **当前保守行为**：部分付款状态整单拒（`status_not_committable`）；跨币种即拒（`currency_mismatch`），无折算。
- **影响面**：状态白名单、`payment_status` / `paid_amount`、多币种折算。
